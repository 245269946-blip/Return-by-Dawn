# 《逾期之书》项目骨架与框架落地方案

> 状态：待确认（基于 M1 v0.9 引擎 + v6 定稿 + 道具落地清单）
> 目标：先把"项目骨架 / 框架怎么落"想清楚并确认；内容深化的 3/4/5 暂停。
> 本方案只回答三件事——**点击探索 + 区域切换 + 对话系统**如何完整呈现我们想讲的内容。

---

## 0. 一句话结论

现有的 M1 v0.9 引擎已经实现了"你设想的每夜入口结构"约 80%：
`notice → enter → hub → scene → return → reveal → ending`
就是"通知 → 进入 →（章节式入口）自由探索 → 收束 → 三选一"。

我们要做的**不是重造引擎**，而是在它之上**长出三样东西**：

1. **点击探索**：场景内多点可点（热点），每个物件只给"一句话"，玩家自己脑补。
2. **区域切换**：把现有 `hub` 升级为"区域图 + 跨夜 meta 入口"；单夜内是区域切换，夜之外是通知箱/背包/回溯（延后）。
3. **对话系统**：管理员**常驻反应层**——这是和锈湖最大的区别，也是情绪核心。

你提的"通知 → 点击 → 展开（日期/书名/逾期原因）→ 进入区域"，与现有 `notice` 节点完全同构；只是把"按钮"换成"可展开的通知卡"。这一点属于**核心循环升级，不阻塞**。

你提的"背包 / 拼起来 / 回溯系统"是**夜之外的一层**（meta 层），建议等三件套验证稳了再建，避免先在 meta 上过度设计而核心体验没落住。

## 2. 三套核心系统的数据模型

### 2.1 点击探索（Click Exploration）= 场景热点 + Florence 一句话

**模型升级**：`scene` 从"一段描写 + 一个谜题"，升级为"**容器 of hotspots（可点物件）**"。

每个 hotspot 点击后产生三件事：
1. **物件自身的一句话**（不解释、留白，玩家脑补）；
2. **触发一条管理员反应**（见 2.3，常驻在场）；
3. **可选**：解锁一条线索 / 推进一个谜题 / 触发一个状态钩子。

```js
// 场景 = 容器，内含多个可点物件
const SCENES = {
  reading_room: {
    name: '阅览区',
    desc: '一排书脊上烫着被拆散的字。',   // 进场氛围（一句）
    hotspots: {
      lamp: {                                  // 物件 = 热点
        label: '一盏旧灯',
        once: '这盏灯为什么一直亮着？',       // Florence 式：一句话，不解释
        again: '灯还亮着。和刚才一样。',
        curator: '因为以前有人怕黑。',         // 管理员反应（见 2.3）
        onCuratorAsk: '谁？→ 忘了。',          // 追问链（其实说自己）
        unlocks: 'c_lamp',                     // 可选：线索/钩子
      },
      phone: {
        label: '旧手机',
        once: '联系人：妈妈。最近通话：8个月前。', // 不给结论，玩家自己懂
        curator: '……（管理员没接话，只是看你）',
      },
      spine: { /* 书脊拆字谜题，answer 来自内容 */ },
    },
    exits: { '走向旧书架': 'shelf_area', '到归还台': 'return_desk' },
  },
};
```

**写作纪律（对齐 §11.1 + 你第二个点）**：
- 物件给**事实**，不给**结论**。不写"主角因为家庭关系疏远所以……"。
- 玩家自己意识到："等等，这不是别人吗？为什么这么像我？"——这是魅力来源。
- 文字占比可以很高，但形态是**物件→一句话→脑补**，不是大段小说。

### 2.2 区域切换（Area Switching）= 区域图 + 跨夜 meta 入口

**单夜内**（已有，升级形态）：
- `hub` 现在是"按钮列表式"区域切换。升级为**可点地图 / 通道**更"锈湖"：点击墙上的门、窗边的通道、柜台后的楼梯来切换区域，而非一列按钮。
- 每个区域带**展示隐喻**（来自道具落地清单 §1）：借阅台=程序性温柔的入口、书库深处=漏雨的裂缝、门廊=进出的阈限……区域本身就是叙事。

**跨夜 meta（延后，你提的通知箱/背包/回溯）**：
- 这是**夜之外的一层**：一个"首页 / 通知箱"模块，积累每一夜不同的逾期通知、每一夜的物品、以及"拼起来"的动作。
- 骨架里先留接口（`meta.js`），三件套验证完再建。
- **通知作为每夜入口**：`notice` 节点升级为"可展开通知卡"（日期 / 书名 / 逾期原因），点击展开→进入。这是核心循环内的事，不依赖 meta 层。

### 2.3 对话系统（管理员常驻反应层）= 与锈湖最大的区别

**核心判断**：锈湖 = 玩家孤独探索；《逾期之书》 = 玩家探索，但身边有一个人。
因为情绪核心不是谜，是**陪伴**。

**模型**：每个交互（点物件 / 解谜 / 抉择）除自身结果外，额外触发一条**管理员 reaction**。

reaction 三用（对齐 §13 管理员职能，但升为"系统层"而非"描述"）：
1. **旁白**：描写空间、落一句于沉默处，不在热闹处抢戏；
2. **镜子**：把你刚才的选择平静映回（"你把它放回去了。和上次一样。"），不评判；
3. **渐进揭示**：前夜只露微小异常（无借出记录 / 想不起自己名字），为后期揭晓埋缝。

**常驻 UI**：底部一条"管理员对话条"，始终在，但不占主叙事区。多数时候是一两句，且常"说了像没说"——

> 玩家点旧灯："这盏灯为什么一直亮着？"
> 管理员："因为以前有人怕黑。"
> 玩家："谁？"
> 管理员："忘了。"（其实他说的是自己。）

**数据模型**：每个 hotspot / action 可带 `curator` 字段；另设全局 `onEvent` 钩子，让特定交互（如三选一、知被遗忘）统一触发管理员的"镜子/揭示"反应。

## 3. 两个设计点的判断（你问有没有道理）

### 3.1 管理员一直在场 ✅ 完全同意，且应定为骨架第一原则

- 对齐 §13，但 §13 把它当"职能描述"；骨架要把它当**系统层**——**每个交互都过一遍管理员反应**，而非偶尔出现。
- **关键纪律（防过度）**：在场 ≠ 话多。管理员是"陪伴"，不是"全程解说"。默认反应应是"沉默 / 一句"，只有特定物件或钩子才开口。这和你第二个点（不解释）自洽。
- 例：玩家点"旧手机→联系人妈妈→8个月前"，管理员**不接话、只是看**——这种"在场但不打断"比台词更有重量。

### 3.2 文字探索 = 物件→一句话→脑补 ✅ 同意，且应定为内容写作纪律

- 对齐 §11.1 行文语气 + 道具落地清单"展示隐喻"。
- 明确禁止：大段心理分析、作者旁白替玩家总结、把"TA 是你"当抖包袱。
- 这是"文字占高比例但不做成大段小说"的落地方式——文字量在，但**密度低、留白高**，靠玩家的脑补完成。

## 4. 提议的项目骨架（文件结构）

```
overdue-book/
  engine/                 # 通用引擎（升级版，内容无关）
    engine.js             # 状态机 + 渲染调度（接管三系统）
    companion.js          # 管理员反应层（对话系统 · 2.3）
    hotspots.js           # 通用热点/物件交互（点击探索 · 2.1）
    meta.js               # 跨夜层接口：通知箱+背包+回溯（后续建，先留接口）
  content/
    nights/night_a.js     # 单夜内容（纯数据：scenes/hotspots/curator hooks）
    regions.js            # 区域总览 + 展示隐喻（道具清单 §1）
    props.js              # 道具/物证定义（道具清单 §3/§4）
  ui/
    index.html
    style.css
    components/           # notice-card / companion-bar / hotspot-layer
  demo/                   # 保留旧 M1 作对照（不删）
```

**原则延续（§5）**：引擎与内容分离。加一夜 = 加一个 `nights/*.js`，零改引擎。
**新增原则**：内容与视图分离——`night_*.js` 导出**纯数据对象**（场景/热点/馆长钩子），引擎负责渲染成热点层/对话条/区域图。

## 7. 待确认 / 我的两点碰撞

**碰撞 1（关于 meta 层）**：背包/回溯/通知箱是"夜之外"的一层，建议**先验证三件套再建**。否则容易在 meta 上过度设计，而核心体验没落稳。你也说"这个可以后面再说"——我同意，且主张它不该阻塞骨架确认。

**碰撞 2（关于"一直在场"）**：我完全认同"管理员一直在场"是骨架第一原则，但必须同时立一条反向纪律——**在场 ≠ 话多**。管理员默认应是"沉默 / 一句"，只在特定物件（如旧灯）和特定钩子（如三选一、知被遗忘）才开口。一旦变成"每点一下都冒一句解说"，陪伴感会塌成说教感，也和你"不解释、让玩家自己懂"的第二个点自相矛盾。

1. 区域切换的 UI 形态——你更想要「可点地图/通道」还是「按钮列表（先快后精）」？

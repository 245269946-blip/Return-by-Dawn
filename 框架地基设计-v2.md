## 二、空间模型 v2.2（真实图书馆动线 · 严格楼层树 · 不乱飞）

### 2.1 楼层树（邻接＝走门/走廊，不能跨区跳）

```
入口大门（雨夜门廊）entry_porch        ← 每晚抵达/离开的阈限，伞在此
   │  （唯一出口：只连服务台）
服务台 service_desk                    ← 中枢 hub：登记册/抽屉收便签/失物招领信箱(近景)
   ├── 管理员区①·休息室/背后楼梯  lounge_stairs      [锁定·夜D 解锁]
   ├── 管理员区②·档案室/灯控室    archive_lamp       [锁定·夜D/H 解锁]
   ├── 便民配套区 utility_zone        ← 打印复印/储物柜/失物招领/已还书目墙(近景·跨夜累计)
   └── 阅览区 reading_room           ← 服务台分出的支线
          ├── 书库深处 stacks_deep          ← 漏雨裂缝：旧事书/雨水盆/书架移位(近景)
          └── 自习学习区 study_zone         ← 静音自习位；管理员留灯仪式落点(机制④)
```

- **入口只连服务台**（进/出都得过服务台）——天然锁死「乱飞」。
- **服务台是 home**，永远可回（「⌂ 回到服务台」）。
- 书库深处 / 自习学习区只连阅览区；便民配套区只连服务台；管理员两室只连服务台且夜D 前灰显锁定。
- 全树无跨区捷径（阅览区不再直连门廊、书库不再直连门廊）。

### 2.3 区域注册表（6 基区 ＋ 2 管理员区 ＝ 8 区，对齐 v6 原则14）

- **基区（6）**：`entry_porch` / `service_desk` / `reading_room` / `stacks_deep` / `study_zone`(自习学习区) / `utility_zone`(便民配套区)

> 注：v6 锁「6＋2 共 8 区不可换」与「第 9 空间 meta」，未列全 6 基区名。本表 4 个基区名来自 v6 叙事点名；另 2 个（自习学习区 / 便民配套区）为本版按物理逻辑 + 叙事机制锚定的提案。

### 2.4 两个升区是怎么选出来的（4 候选 → 2 升区 + 2 近景，一把尺子量到底）

> 用户给 4 候选，硬约束只剩 2 基区空位。裁定原则：**谁最锚定 v6 核心机制、且物理上无法被现有区近景化，谁升区；其余降级近景**。

- **升区① `study_zone` 自习学习区**
  - **v6 依据 / 机制锚定**：直接承接羁绊机制④「微小往复仪式（留物/手势/调灯回习惯位）」——用户明说"这是日常来阅读、管理员给你留灯的地方"。留灯是跨夜持久羁绊的核心可视化（常亮灯位），必须有个常驻区承载，且物理上是真实图书馆的静音自习位，不能被服务台/阅览区近景吞掉（它是独立功能空间）。
  - **隐喻**：管理员"没逃、留下扛"的温柔具象；常亮灯位 = 跨夜存在感锚点。
- **升区② `utility_zone` 便民配套区**
  - **v6 依据 / 机制锚定**：decoy 结构需要"大量疑似别人的线索"容器。失物招领（别人遗落物）、储物柜（别人暂存货）、打印复印区天然聚集"别人"痕迹，是前期"帮别人还书/解谜"decoy 的最佳安放处；同时承接跨夜累计的**已还书目墙**（作为该区社区展示墙近景）。物理上是真实图书馆门厅标配，独立成区合理。
  - **隐喻**："别人的人生"碎片集散地，前期读成别人、后期回照成自己。

- **降级近景① 旧书区 → `reading_room`**
  - 与阅览区是"普通阅览 vs 旧书特藏"的区隔，物理上就是阅览厅里一道旧书专区隔断，**可用阅览区近景实现**，不必拔成第 7 基区。
- **降级近景② 多功能电脑区 → `service_desk`**
  - 公共检索/上机位是服务台旁的标配辅助功能，**可用服务台近景实现**，不必独立成区。

> 一致性声明：本版否掉的"还书归档陈列区/流动书架区"（v2.1 自编）也按同一标准降级为近景（见 §二.5），**不是只否我的、留用户的**——尺子对所有人一致。

## 三、场景层级（大场景→近景→结算，骨肉对应）

- **大场景（8 区＋第9空间）**：地点本身 ＋ 隐喻 ＋ 管理员 enter 反应。现有 `region` 节点承载。
- **近景 closeup**：zoom 进物件/子场景（信 / 借书卡 / 灯 / 伞 / 失物招领信箱 / 登记册 / 雨水盆 / 已还书目墙 / 旧书专区 / 公共检索机），可递归子热点。现有 `closeup` 节点承载。
- **结算 settlement**：努力动作 / 不在场 / 关键物证 的专属反馈页。现有 `settlement` 节点承载。

三层用法本设计只**规范**，不改现状节点结构。
